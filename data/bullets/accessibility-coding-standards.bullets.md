- Strive to make websites and digital assets accessible to the widest possible audience.
- Consider accessibility needs when architecting the site.
- Use best practices to ensure that content can be navigated and read by all.
- Build accessibility reviews into existing processes to ensure code is accessible.
- Provide text alternatives (alt text) for non-text content, such as images, charts and graphs.
- Provide accurate transcripts, captions, and descriptions for video.
- Ensure color choices have sufficient contrast for users with visual impairments.
- Test to ensure the website is keyboard-navigable.
- Use semantic HTML tags, such as `<h1>`, `<ul>`, and `<button>` to structure content for assistive technologies.
- Default to using HTML whenever possible and use WAI-ARIA modestly.
- Check your work with automated checkers and manually test new interactive components.
- Public-facing code should strive to meet Web Content Accessibility Guidelines (WCAG) 2.2 AA.
- Author-facing interfaces should strive to meet the Authoring Tool Accessibility Guidelines (ATAG) 2.0.
- Use WAI-ARIA attributes to add semantic meaning to markup and markup patterns.
- Announce changes to content on a page to screen reader users with aria-live.
- Ensure any feature that can be operated using a mouse also works using a keyboard.
- Use Drupal's "visually-hidden" CSS class to hide content visually, but leave it available to screen readers.
- Use Drupal's "visually-hidden" CSS class with the "focusable" class to make content visible when a keyboard user focuses on it.
- Use Drupal’s CSS class “hidden” to hide content from both sighted users and from screen readers.
- Highlight form errors next to the input field with Drupal's inline form errors module.
- Use a <fieldset> with a <legend> element to make visually grouped form inputs machine-readable.
- Use animation responsibly and respect user preferences.
- Test your work with automated accessibility checkers.
- Test that your module/theme is keyboard accessible.
- Test your work with assistive technology like the NVDA Screen Reader or Apple’s VoiceOver.
- When testing the accessibility of your forms, ensure that a non-visual user will be alerted to the error.
